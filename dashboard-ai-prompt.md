# Prompt: Гүйлгээний Dashboard + Live AI ангилал баталгаажуулалт

## Зорилго

Бүртгэгдсэн банкны гүйлгээг харах, шүүх **хялбар, түргэн dashboard** болон **live AI ангилал** механизм бүтээ. Танигдаагүй мерчантыг AI таамаглаж, хэрэглэгч баталгаажуулна — баталгаажуулсан зүйл "learned override" болж дараа автоматаар хэрэглэгдэнэ. Бүх UI **утаснаас ажиллахаар responsive** байх ёстой.

## Контекст — одоо байгаа систем

Бэлэн ажиллаж байгаа:
- Listener (IMAP IDLE) → parse (`parsers/golomt.js`) → `categorize()` → API руу push → DB
- API: Node.js + Express, `POST /api/transactions`, `GET /api/transactions` (шүүлттэй)
- DB: `transactions` хүснэгт (message_id, amount, currency, txn_date, description, type, category, account_last4, raw, created_at)
- `categorize()` нь дүрэмд суурилсан — одоо ~39% ангилж, 61% нь "other" (Голомтын 16 тэмдэгтэд таслагдсан таниулашгүй мерчант кодууд: STOREBOM, THE LBOM гэх мэт)
- DB-д аль хэдийн 1000+ гүйлгээ байгаа

## Технологийн стек (санал)

- **Frontend:** React + Vite + Tailwind CSS (responsive хялбар, хүснэгтэд тохиромжтой). Хүнд framework хэрэггүй — энэ дотоод хэрэгсэл.
- **Backend:** одоо байгаа Express API-г өргөтгөнө (шинэ сервер биш).
- **AI:** Anthropic Claude API (categorize fallback-д). API key env-ээс.
- **Charts (хэрэв нэмбэл):** хөнгөн сан (recharts) — гэхдээ эхэнд хүснэгт/шүүлтүүр гол, график хоёрдогч.

## Хэсэг 1: Dashboard (хялбар, түргэн)

### 1.1 Гүйлгээний хүснэгт
- Гүйлгээг хүснэгтээр харуулна: огноо, тайлбар (description), дүн, төрөл (зарлага/орлого), ангилал, данс.
- Зарлага улаан/орлого ногоон гэх мэт өнгөөр ялгах.
- Pagination эсвэл infinite scroll (1000+ мөр тул бүгдийг нэг дор ачаалахгүй).
- Хамгийн сүүлийн гүйлгээ дээр.

### 1.2 Шүүлтүүр (гол функц)
- Огнооны хүрээ (from–to)
- Ангилал (dropdown, олон сонголт)
- Төрөл (зарлага / орлого / бүгд)
- Текст хайлт (description-д)
- Дүнгийн хүрээ (min–max) — сонголт
- Шүүлтүүр API-ийн `GET /api/transactions` query параметрээр ажиллана (frontend дээр бүгдийг ачаалж шүүхгүй — backend шүүнэ).

### 1.3 Хураангуй (хөнгөн)
- Шүүсэн үр дүнгийн: нийт зарлага, нийт орлого, гүйлгээний тоо.
- Ангиллаар задаргаа (хялбар: ангилал бүрийн нийт дүн, хувь). Энэ хэсэгт жижиг pie/bar chart нэмж болно, гэхдээ заавал биш.

### 1.4 Responsive (заавал)
- Утасны дэлгэцэнд (~380px) хүснэгт уншигдахуйц байх: жижиг дэлгэцэд карт хэлбэрт шилжих эсвэл хэвтээ scroll-той бай.
- Шүүлтүүр утсан дээр эвхэгддэг (collapsible) панел байж болно.
- Tailwind-ийн responsive breakpoint ашиглана.

## Хэсэг 2: Live AI ангилал баталгаажуулалт (гол шинэ функц)

### 2.1 Урсгал
1. Шинэ гүйлгээ ирэхэд listener эхлээд `categorize()` (дүрэм + learned overrides) -оор ангилахыг оролдоно.
2. **Олдвол:** шууд тэр ангилалаар хадгална (`status: 'classified'`).
3. **Олдохгүй (танигдаагүй мерчант):**
   - AI (Claude API)-д тайлбарыг өгч, боломжит ангилал + итгэлийн түвшин асуана.
   - Гүйлгээг `status: 'pending_review'`, `ai_suggested_category`, `ai_confidence` талбартай хадгална.
   - Dashboard-ийн "Баталгаажуулах" хэсэгт гарч ирнэ.
4. Хэрэглэгч баталгаажуулах/засах → learned override болж хадгалагдана → гүйлгээний `category` шинэчлэгдэж, `status: 'classified'` болно.
5. **Цаашид тэр мерчант автоматаар** learned override-оор ангилагдана — дахин асуухгүй.

### 2.2 AI асуулт (Claude API)
- Backend дээр endpoint: `POST /api/ai-categorize` — тайлбар (description) авч, боломжит ангиллыг буцаана.
- Claude API-д өгөх prompt: одоо байгаа ангиллын жагсаалтыг (categories.config-оос) өгч, "энэ мерчант тайлбар аль ангилалд хамаарах вэ? Магадгүй бол хамгийн ойрыг сонго, мэдэхгүй бол 'Бусад' гэж хариул" гэж асуана. JSON хэлбэрээр ангилал + итгэлийн түвшин (low/medium/high) буцаахыг шаард.
- **Анхаар:** Голомтын мерчант код 16 тэмдэгтэд таслагдсан, товчилсон тул AI ч мэдэхгүй байж магадгүй. Тийм үед AI "Бусад" эсвэл low confidence буцаах ёстой — буруу таамаглахаас болгоомжил. Энэ тохиолдлыг prompt-д тодорхой зааж өг.
- AI key, rate limit-г env-ээр удирд. AI дуудалт амжилтгүй бол гүйлгээ "other"-оор үлдэж, дараа дахин оролдох боломжтой бай (систем унтрахгүй).

### 2.3 Баталгаажуулах UI
- Dashboard дээр "Баталгаажуулах хүлээж буй" (`pending_review`) гүйлгээний жагсаалт.
- Тус бүр дээр: тайлбар, дүн, AI-ийн санал болгосон ангилал (итгэлийн түвшинтэй), ангилал сонгох dropdown.
- Хэрэглэгч: AI саналыг **зөвшөөрөх** (нэг товч) эсвэл **өөр ангилал сонгох**.
- Баталгаажуулсны дараа тэр мерчантын бүх ижил гүйлгээнд хэрэглэх эсэхийг асуух (жишээ: "STOREBOM-ийн бүх 12 гүйлгээг 'Хүнс' болгох уу?"). Ингэснээр нэг баталгаажуулалт олон мөрийг засна.

### 2.4 Learned overrides хадгалалт
- Шинэ DB хүснэгт: `category_overrides` (id, merchant_pattern, category, created_at).
- `categorize()`-г шинэчилж, дүрэм шалгахаас **өмнө** learned overrides-ийг шалгадаг болгоно (override эхэнд).
- Override-ийг merchant pattern-аар тааруулна (exact эсвэл normalize хийсэн substring). Жишээ: `STOREBOM` → "Хүнс".
- Listener болон одоо байгаа categorize урсгал хоёулаа энэ override-ийг ашиглана.

## Хэсэг 3: API өргөтгөл

Одоо байгаа Express API-д нэмэх endpoint-ууд:
- `GET /api/transactions` — шүүлттэй жагсаалт (огноо, ангилал, төрөл, хайлт, pagination). Аль хэдийн байгаа бол өргөтгө.
- `GET /api/transactions/pending` — баталгаажуулах хүлээж буй гүйлгээ.
- `GET /api/summary` — хураангуй (шүүлттэй: нийт зарлага/орлого, ангиллаар задаргаа).
- `POST /api/ai-categorize` — AI ангилал санал (дотоод, listener эсвэл dashboard дуудна).
- `PATCH /api/transactions/:id/category` — гүйлгээний ангилал засах (баталгаажуулах). Body-д `applyToAll: true/false` (тэр мерчантын бүгдэд хэрэглэх эсэх).
- `POST /api/overrides` — learned override нэмэх.

Бүх endpoint одоо байгаа auth (API key) -аар хамгаалагдана. Хувийн санхүүгийн өгөгдөл тул заавал.

## Хэсэг 4: DB өөрчлөлт (миграци)

`transactions` хүснэгтэд нэмэх:
- `status` TEXT DEFAULT 'classified' ('classified' | 'pending_review')
- `ai_suggested_category` TEXT (nullable)
- `ai_confidence` TEXT (nullable: 'low'|'medium'|'high')

Шинэ хүснэгт:
- `category_overrides` (id, merchant_pattern TEXT UNIQUE, category TEXT, created_at)

Миграцийг SQL файлаар гарга, одоо байгаа 1000+ мөрийг алдагдуулахгүй (ALTER TABLE, default утгатай).

## Цаашдын бодолцол — утаснаас ажиллах (одоо хэрэгжүүлэхгүй, бэлэн бай)

Энэ системийг ирээдүйд утаснаас бүрэн ашиглахаар тооцож архитектур зохио (одоо зөвхөн responsive web хийнэ, доорхыг бэлтгэл болгон бодолцоно):
- API-г frontend-ээс тусдаа, цэвэр REST байлгах (дараа mobile app ижил API-г ашиглана).
- Баталгаажуулах үйлдлийг push notification-д тохирохоор бод (дараа: гүйлгээ ирмэгц утсанд "энэ мерчантыг ангилна уу?" мэдэгдэл).
- Auth-г token-д суурилсан байлгах (дараа mobile-д хялбар).
- Эдгээрийг ОДОО хэрэгжүүлэхгүй, зүгээр л код/архитектурыг тэдгээрт саад болохооргүй цэвэр бай. README-д "ирээдүйд mobile/push нэмэх боломжтой" гэж тэмдэглэ.

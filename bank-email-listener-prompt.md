# Prompt: Банкны имэйл real-time бүртгэлийн listener service

## Зорилго

Gmail хайрцагт банкнаас гүйлгээний имэйл ирмэгц **бараг real-time** (секундын дотор) илрүүлж, агуулгыг задлан, ангилж, гадаад вэбсайтын API руу POST хийдэг, **байнга ажилладаг** Node.js backend service бичнэ үү. Систем нэг ч имэйл алдахгүй, давхардуулахгүй, чимээгүй унтрахгүй байх ёстой.

## Технологийн стек

- **Хэл:** Node.js (ES modules, async/await)
- **IMAP:** `imapflow` (IDLE горим дэмждэг)
- **Имэйл parse:** `mailparser` (`simpleParser`), HTML задлахад `cheerio`
- **OAuth2:** `google-auth-library` (Gmail-д XOAUTH2-аар нэвтрэх)
- **DB:** SQLite (`better-sqlite3`) — UID болон боловсруулсан Message-ID хадгалах
- **Process manager:** `pm2`-д тохирсон байх (ecosystem.config-той)
- **Логгер:** энгийн structured logging (`pino` эсвэл console-д timestamp-тай)

## Архитектур / урсгал

```
Банк → Gmail хайрцаг → IMAP IDLE listener (сонсох)
  → шинэ имэйл илрэх → банкны хаягаар шүүх
  → parse (дүн, огноо, тайлбар салгах) → ангилах
  → давхардал шалгах → вэбсайтын API руу POST → DB-д бүртгэх
```

## Заавал хэрэгжүүлэх шаардлагууд

### 1. Найдвартай ажиллагаа (Reliability)

- **Reconnect (exponential backoff):** Холболт тасрах, алдаа гарах бүрд автоматаар дахин холбогдоно. Хүлээх хугацааг 1с → 2с → 4с → ... дээд тал нь 60с хүртэл нэмэгдүүлэх. Амжилттай холбогдвол backoff-ийг reset хийнэ.
- **Access token refresh:** OAuth access token ~1 цагт дуусдаг. Token дуусахаас **өмнө (50 минут тутамд)** холболтыг автоматаар дахин эхлүүлж шинэ access token авна. Timer-ээр удирдана. Энэ байхгүй бол 1 цагийн дараа чимээгүй унтарна — заавал хэрэгжүүл.
- **lastSeenUid-г DB-д хадгалах:** Хамгийн сүүлд боловсруулсан имэйлийн UID-г SQLite-д хадгална. Санах ойд хадгалж болохгүй.
- **Catch-up:** Сервис асахдаа DB-д хадгалсан UID-ээс хойших бүх имэйлийг гүйцэж боловсруулна (унтарсан хугацаанд ирсэн имэйлийг алдахгүй). UIDVALIDITY өөрчлөгдсөн тохиолдлыг зохицуулна.

### 2. Идэмпотентность (давхардал хамгаалалт)

- Имэйл бүрийн `Message-ID`-г DB-д хадгална. Боловсруулахаас өмнө "энэ Message-ID орсон уу?" гэж шалгана. Орсон бол **алгасна**.
- Энэ нь reconnect, catch-up, давхар event-ийн үед гүйлгээ давхар бүртгэгдэхээс хамгаална.

### 3. Имэйл шүүлт ба parse

- **Шүүлт:** Зөвхөн `alert@golomtbank.com` хаягаас ирсэн имэйлийг боловсруулна. Бусдыг алгасна. Хаягийг `BANK_SENDER` env хувьсагчаар уншина (default утга нь `alert@golomtbank.com`), ингэснээр кодод хатуу бичигдэхгүй.
- **Нэг parser:** Энэ систем зөвхөн Голомт банкны (`alert@golomtbank.com`) имэйл боловсруулна. Тиймээс олон банкны registry/strategy загвар **хэрэггүй** — нэг `parseGolomt()` функц хангалттай. Гэхдээ функцийг тусдаа `parsers/golomt.js` модульд гаргаж, ирээдүйд өөр банк нэмэхэд хялбар байхаар цэвэр интерфэйстэй (имэйлийн агуулга авч, нэгдсэн transaction object буцаадаг) бай.
- **HTML дэмжих:** Голомт банк гүйлгээний имэйлээ HTML хүснэгтээр илгээх магадлал өндөр. `parsed.html`-ийг `cheerio`-оор задлах боломжтой бай. Plain text (`parsed.text`) fallback-тэй.
- **Гаргаж авах талбарууд:** дүн (amount), валют, огноо (date), тайлбар/мерчант (description), гүйлгээний төрөл (зарлага/орлого), дансны сүүлийн оронтой дугаар (боломжтой бол).
- **Parse алдсан үед:** Чимээгүй алдахгүй. Шаардлагатай талбар (дүн г.м) олдохгүй бол алдааг log хийж, имэйлийг DB-д `status: 'parse_failed'` гэж тэмдэглэ. Гэхдээ Message-ID-г бүртгэж, дахин оролдохгүй (эсвэл retry queue-д оруулах сонголттой).

### 4. Ангилал

- Энгийн дүрэмд суурилсан (keyword/regex) ангилагчаар эхэл: тээвэр, хүнс, бэлэн мөнгө, төлбөр, бусад г.м. Категориудыг тусдаа config файлд тавь (keyword → category mapping), кодод хатуу бичихгүй.
- Дараа нь AI ангилал залгах боломжтой байхаар интерфэйс цэвэр бай (одоохондоо хэрэгжүүлэхгүй, зүгээр л өргөтгөх боломжтой бүтэцтэй бай).

### 5. Вэбсайт руу илгээх (API push)

- Гүйлгээг тохируулсан endpoint URL руу POST хийнэ (JSON).
- **Authentication:** Header-т API key эсвэл HMAC гарын үсэг нэмнэ (env-ээс уншина).
- **Retry:** POST амжилтгүй бол exponential backoff-той 3-5 удаа дахин оролдоно. Бүгд амжилтгүй бол DB-д `status: 'push_failed'` гэж тэмдэглэж, дараа дахин илгээх боломжтой бай (DB-д "илгээгдээгүй" гүйлгээг scan хийж re-push хийдэг функц).
- **Идэмпотентность түлхүүр:** POST бүрд Message-ID-г idempotency key болгон явуул, ингэснээр API тал давхардлыг хоёр дахь хамгаалалтаар таслана.

### 6. Аюулгүй байдал

- Бүх нууц утга (`GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `GMAIL_REFRESH_TOKEN`, `WEBSITE_API_KEY`, имэйл хаяг, endpoint URL) **зөвхөн environment variable-ээс** уншина. Кодод хатуу бичихгүй.
- `.env.example` файл гарга (бодит утгагүй).
- `.gitignore`-д `.env`, `*.sqlite`, `node_modules` оруул.
- OAuth scope-ийг боломжит хамгийн бага эрхээр тохируул (имэйл унших эрх; устгах/илгээх эрхгүй). IMAP-д `https://mail.google.com/` шаардлагатай бол тэмдэглэл бич.

### 7. Хяналт ба алдаа мэдээлэх (Monitoring)

- Бүх чухал үйлдлийг (холбогдсон, имэйл ирсэн, parse, push, алдаа, reconnect) timestamp-тай structured log-оор бич.
- **Heartbeat:** Тодорхой хугацаа (тохируулж болох) бүрд "alive" log/тэмдэглэгээ бич. Сүүлийн имэйлээс хэт удвал warning гарга.
- **Алдааны мэдэгдэл (optional hook):** Чухал алдаа (дараалсан reconnect амжилтгүй, token авч чадахгүй г.м) гарвал дуудагдах `notifyError()` функцийг тусад нь гарга. Одоохондоо зүгээр л log хийдэг байж болох ч, дараа Telegram/имэйл залгахад бэлэн интерфэйстэй бай.

### 8. Байршуулалт ба бүтэц

- Codebase-ийг цэвэр модуль болгож хуваа:
  - `imap-client.js` (холболт, IDLE, reconnect, token refresh)
  - `parsers/golomt.js` (Голомт банкны имэйл parser — нэг банк тул registry хэрэггүй)
  - `categorize.js`
  - `db.js` (SQLite: UID, processed Message-IDs, failed transactions)
  - `push.js` (API руу илгээх + retry)
  - `config.js` (env унших, валидаци)
  - `logger.js`
  - `index.js` (бүгдийг холбож ажиллуулах)
- **Эхний нэг удаагийн OAuth refresh_token авах туслах скрипт** (`scripts/get-token.js`) — localhost callback серверээр consent урсгал хийж, refresh_token-г хэвлэнэ.
- `pm2`-д зориулсан `ecosystem.config.cjs` гарга (autorestart, max memory restart, log файл).
- `README.md`-д: суулгах, OAuth тохируулах (Google Cloud дээр project үүсгэх, Gmail API enable хийх, consent screen, client ID авах алхамууд), `.env` бөглөх, `get-token.js` ажиллуулах, `pm2`-оор асаах заавар бич.

## Чанарын шаардлага

- Бүх async үйлдэл try/catch-тэй, ямар ч алдаа listener-ийг бүхэлд нь унтраахгүй (нэг имэйлийн алдаа бусдыг зогсоохгүй).
- Кодыг уншихад ойлгомжтой, MN/EN тайлбар comment-тэй бай.
- `parseTransaction`, `categorize`-д unit test (жишээ имэйл агуулга дээр) хэдийг нэмбэл сайн.
- Edge case-уудыг тооцоо: хоосон имэйл, parse алдсан, API унтарсан, token дууссан, UIDVALIDITY өөрчлөгдсөн, сервис дунд унтарсан.

## Чухал тэмдэглэл агентад

- Энэ систем **зөвхөн Голомт банк** (`alert@golomtbank.com`)-нд зориулагдсан. Олон банкны registry бүтэц битгий хий — нэг `parseGolomt()` parser хангалттай.
- Голомт банкны имэйлийн **яг формат тодорхойгүй** тул `parsers/golomt.js`-ийг TODO тэмдэглэгээтэй, бодит имэйл орж ирэхэд хялбар бөглөх боломжтой бүтэцтэй гарга. Жишээ regex-уудыг ерөнхий байдлаар (дүн: `[\d,]+`, валют: `₮|MNT|төгрөг`, огноо: `YYYY-MM-DD`) тавьж, comment-д "энд Голомт банкны бодит формат тааруулна" гэж тэмдэглэ. HTML хүснэгт байх магадлал өндөр тул `cheerio`-оор задлах жишээ бэлд.
- `index.js`-ийг ажиллуулахад бэлэн, харин `parsers/golomt.js` болон config-ийг хэрэглэгч бодит имэйлийн форматад тааруулж бөглөх шаардлагатай гэдгийг README-д тодорхой бич.

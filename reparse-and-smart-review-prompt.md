# Prompt: Дахин parse + шалтгаан тэмдэглэл + ухаалаг (төрөл-мэдрэмжтэй) баталгаажуулалт

## Контекст

Голомт банкны гүйлгээ бүртгэгч систем ажиллаж байна (listener → parse → categorize → API → DB → dashboard). Хоёр шинэ шаардлага гарсан:

1. Parser засагдсан (огнооны 2 загвар, олон мөрт тайлбар) тул хуучин имэйлүүдийг дахин parse хийх хэрэгтэй.
2. Одоогийн ангилал хэт ерөнхий — хэрэглэгч "яагаад мөнгө гарсан" шалтгааныг ойлгож чадахгүй байна. Гүйлгээний төрлөөс хамаарч өөр өөр мэдээлэл асуух ухаалаг систем хэрэгтэй.

---

## Хэсэг 1: Хуучин имэйлийг дахин parse хийх

**Зорилго:** Засагдсан `parsers/golomt.js`-ээр DB дэх бүх хуучин гүйлгээг дахин боловсруулж, "-" (null) болсон талбаруудыг (огноо, тайлбар, данс) нөхөх.

**Хэрэгжүүлэлт:**
- Нэг удаагийн скрипт: `scripts/reparse.js`.
- Gmail-аас `alert@golomtbank.com`-ийн бүх имэйлийг дахин уншиж (эсвэл хадгалсан raw байвал түүнээс), засагдсан `parseGolomt`-оор дахин задлана.
- DB дэх гүйлгээг `messageId`-аар тааруулж, **дутуу талбаруудыг шинэчилнэ** (UPDATE). Шинээр insert хийхгүй — давхардуулахгүй.
- Хэрэглэгч аль хэдийн гараар оноосон зүйлийг (note, friendly_name, баталгаажуулсан category) **дарж бичихгүй** — зөвхөн parser-аас гарах түүхий талбаруудыг (date, description, account, amount, balance) нөхнө.
- Скрипт ажилласны дараа тайлан: хэдэн гүйлгээ шинэчлэгдсэн, хэд нь одоо ч дутуу (хэрэв ямар нэг имэйл огт олдоогүй бол).
- Дахин parse хийсний дараа, шинээр задарсан тайлбартай гүйлгээнүүдийг `categorize` дахин ажиллуулж ангилахыг оролдоно.

## Хэсэг 2: Шалтгаан/тэмдэглэл (note) талбар

**Зорилго:** Гүйлгээ бүрд хэрэглэгчийн өөрийн "шалтгаан/тэмдэглэл" хадгалах.

**DB өөрчлөлт (миграци, default-той ALTER):**
- `transactions` хүснэгтэд нэмэх:
  - `note` TEXT (nullable) — хэрэглэгчийн чөлөөт тэмдэглэл (жишээ: "Ээжид сарын мөнгө", "Шулуун дунд хоол идсэн")
  - `is_pos` BOOLEAN (nullable) — POS гүйлгээ эсэх (BOM дүрмээр тодорхойлно)
  - `merchant_place` TEXT (nullable) — POS гүйлгээний танигдсан газрын нэр (жишээ: "Шулуун дун")

**Dashboard:**
- Гүйлгээ бүр дээр note харах, засах боломж (inline edit эсвэл modal).
- Жагсаалтад note баганыг (эсвэл icon) харуулах.

## Хэсэг 3: POS эсэхийг таних (BOM дүрэм)

**Дүрэм:**
- Гүйлгээний `description` нь `BOM`-оор төгссөн (эсвэл `BOM` агуулсан) бол → **POS гүйлгээ** (`is_pos = true`). Картаа тодорхой газар уншуулсан.
  - Жишээ: `ShuluBOM`, `0930 LANN BOM`, `STOREBOM`, `0930 BUJINBOM`.
- `BOM`-гүй бол → POS биш (`is_pos = false`): шилжүүлэг, цахим төлбөр, бэлэн мөнгө, SOCIALPAY гэх мэт.
  - Жишээ: `HER-БАТСАЙХАН ТӨ` (шилжүүлэг), `SOCIALPAY ГҮЙЛГЭЭ` (цахим төлбөр), `Deposit: :949625` (бэлэн мөнгө).

**Хэрэгжүүлэлт:**
- `parsers/golomt.js` эсвэл categorize урсгалд `is_pos`-ийг тооцоолж тогтооно (description-д `BOM` төгсгөл/орц байгаа эсэхээр).
- Анхаар: `BOM` нь үгийн төгсгөлд POS дохио. Худал эерэг (false positive)-оос болгоомжил — жинхэнэ мерчантын нэрэнд "bom" орох магадлал бага ч, дүрмийг үгийн төгсгөлд тааруулбал найдвартай.

## Хэсэг 4: Ухаалаг (төрөл-мэдрэмжтэй) баталгаажуулалт — гол функц

Баталгаажуулах хэсэгт гүйлгээний **төрлөөс хамаарч өөр асуулт** асууна.

### 4.1 POS гүйлгээ (`is_pos = true`)
- Контекст: газар тодорхой (картаа тэнд уншуулсан), зөвхөн ямар газар, ямар ангилал болохыг мэдэхгүй.
- **Асуулт:** "Энэ POS гүйлгээ. Ямар газар вэ?"
- Хэрэглэгчээс авах:
  - **Газрын нэр** (merchant_place) — жишээ: `ShuluBOM` → "Шулуун дун"
  - **Ангилал** — жишээ: "Хоол / Кафе"
  - (Сонголтоор) note — нэмэлт тэмдэглэл
- AI санал: тайлбараас газрыг таамаглаж болох ч (BOM өмнөх хэсэг), таслагдсан код тул болгоомжтой, low confidence. Хэрэглэгч баталгаажуулна.
- Баталгаажуулсны дараа: тэр мерчант код (BOM-ийн өмнөх хэсэг) → газрын нэр + ангилал learned override болж хадгалагдана. Дараа тэр код ирэх бүрд автоматаар газар + ангилал танигдана.

### 4.2 POS биш гүйлгээ (`is_pos = false`)
- Контекст: тодорхой газар биш — шилжүүлэг/төлбөр. "Яагаад, хэнд, юунд" гэдэг нь чухал.
- **Асуулт:** "Энэ шилжүүлэг/төлбөр. Яагаад хийсэн бэ? Хэнд, юунд?"
- Хэрэглэгчээс авах:
  - **Шалтгаан (note)** — нарийн тайлбар. Жишээ: `HER-БАТСАЙХАН ТӨ` → "Ээжид сарын мөнгө явуулсан"
  - **Ангилал** — жишээ: "Гэр бүл / Тусламж"
- AI санал: хүлээн авагчийн нэр (HER-БАТСАЙХАН) -ээс ангилал таамаглахад хэцүү тул асуултыг шалтгаанд чиглүүлнэ. AI зүгээр "энэ шилжүүлэг байна, шалтгааныг тайлбарлана уу" гэж асуух нь дээр.
- Баталгаажуулсны дараа: ижил хүлээн авагч/тайлбартай гүйлгээнд тэр шалтгаан + ангиллыг санал болгох (applyToAll сонголттой).

### 4.3 Ялгааг UI-д тусгах
- Баталгаажуулах карт дээр гүйлгээний төрлийг тэмдэглэх (badge: "POS" эсвэл "Шилжүүлэг/Төлбөр").
- Төрлөөс хамаарч өөр асуулт текст, өөр оролтын талбар харуулна (POS → "Газрын нэр", бусад → "Шалтгаан").

## Хэсэг 5: Ангиллын систем сайжруулах

Одоогийн ангилал хэт ерөнхий тул санхүүгийн дүр зургийг харуулахгүй байна. Сайжруулах:
- Ангилалд "шалтгаан"-ы хэмжээс нэмэх замаар (note талбараар) хэрэглэгч "яагаад мөнгө гарсан"-ыг харах боломжтой болно.
- Dashboard-ийн хураангуйд: зөвхөн ангиллаар биш, боломжтой бол note/газраар бүлэглэх сонголт (жишээ: "Шулуун дунд нийт хэдэн төгрөг зарцуулсан"). Энэ нь хэрэглэгчийн зарлагын жинхэнэ шалтгааныг харуулна.
- `category_overrides` хүснэгтийг өргөтгөх (Хэсэг 4-ийн дагуу): `merchant_pattern`, `category`, `friendly_name` (газрын нэр), мөн POS биш бол `default_note` (нийтлэг шалтгаан) хадгалж болно.

## Чухал тэмдэглэл агентад
- Дахин parse үед хэрэглэгчийн гараар оруулсан note/place/category-г хэзээ ч дарж бичихгүй.
- BOM дүрэм: үгийн төгсгөлд тааруулж false positive-аас сэргийл.
- POS vs шилжүүлгийн асуулт ялгаатай байх нь гол санаа — UI болон AI prompt хоёуланд тусга.
- AI таслагдсан кодыг буруу таамаглахаас болгоомжил — low confidence үед хэрэглэгчээс асуу.
- Одоо байгаа 1000+ гүйлгээ, learned override-уудыг алдагдуулахгүй. Бүх DB өөрчлөлт default-той ALTER.

# Prompt: Системийг ажиллуулах — token-оос эхлээд бүрэн setup

## Контекст

Голомт банк (`alert@golomtbank.com`)-ийн гүйлгээний имэйлийг real-time барьж, parse хийж, ангилж, вэбсайтын API руу хадгалдаг систем бэлэн болсон. Дараах хэсгүүд аль хэдийн бичигдсэн:

- `parsers/golomt.js` — Голомт имэйл parser (тест хийгдсэн, ажилладаг)
- IMAP IDLE listener (reliability, token refresh, reconnect, idempotency-тэй)
- API endpoint (`POST /api/transactions`, идэмпотентность, auth)
- `scripts/get-token.js` — OAuth refresh token авах туслах
- `scripts/collect-descriptions.js` — хуучин имэйлээс тайлбар цуглуулагч
- `categorize.js` + `categories.config.js` — ангилал (одоохондоо таамаг keyword)

Google Cloud дээр аль хэдийн бэлэн:
- "Golomt financial" project, Gmail API enabled
- `golomt-desktop` OAuth client (Desktop app) — Client ID + Secret байгаа
- OAuth consent screen дээр `https://mail.google.com/` scope, test user тохируулсан

## Зорилго

Системийг **тэгээс ажиллуулах хүртэл** дарааллаар нь setup хийж, тус бүрийг шалгах. Доорх алхмуудыг ДАРААЛЛААР гүйцэтгэ. Алхам бүрийг дуусгаад дараагийнх руу орохын өмнө амжилттай болсон эсэхийг баталгаажуул.

> **Чухал:** Зарим алхам (`get-token.js` ажиллуулах) браузер нээж, ХЭРЭГЛЭГЧ өөрөө Google дээр зөвшөөрөл өгөх шаардлагатай. Энэ хэсгийг агент хийж чадахгүй — хэрэглэгчид тодорхой зааварчилж, тэр гүйцэтгэснийг хүлээ.

---

## Алхам 1: Орчны бэлтгэл

1. Listener болон API хоёрын `.env` файл бэлэн эсэхийг шалга. Байхгүй бол `.env.example`-ээс хуулж үүсгэ.
2. Listener `.env`-д дараах байгаа эсэхийг шалга (хэрэглэгч бөглөсөн байх ёстой):
   - `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET` (Google Cloud-аас)
   - `BANK_SENDER=alert@golomtbank.com`
   - `WEBSITE_API_URL`, `WEBSITE_API_KEY`
   - `GMAIL_REFRESH_TOKEN` — ОДОО хоосон байж болно (Алхам 2-т авна)
3. Шаардлагатай бол `npm install` ажиллуулж бүх dependency суулга.
4. Хоосон/дутуу env хувьсагч байвал хэрэглэгчид аль нь дутууг тодорхой хэл.

## Алхам 2: Refresh token авах (хэрэглэгчтэй хамтран)

1. `.env`-д `GOOGLE_CLIENT_ID` болон `GOOGLE_CLIENT_SECRET` тавигдсан эсэхийг баталгаажуул. Үгүй бол хэрэглэгчээс асууж тавиулна.
2. `node scripts/get-token.js` ажиллуул.
3. **Хэрэглэгчид зааварчил:** браузер нээгдэнэ → Google account-аар нэвтэрнэ → "Allow" дарж зөвшөөрөл өгнө → терминалд `refresh_token` хэвлэгдэнэ.
4. Хэрэглэгч `refresh_token`-оо `.env`-д `GMAIL_REFRESH_TOKEN=...` болгож хадгалсныг баталгаажуул.
5. Хэрэв "access_denied" эсвэл "app not verified" алдаа гарвал: хэрэглэгч test user-д өөрийгөө нэмсэн эсэхийг шалгахыг сануул (OAuth consent screen → Test users).

## Алхам 3: API талыг асааж тест хийх

1. API талын DB-г үүсгэж миграци ажиллуул (`migrations/`).
2. API серверийг асаа.
3. `curl`-ээр гурван тест хий, тус бүрийг баталгаажуул:
   - Зөв API key + жишээ гүйлгээ → `201 Created`
   - Буруу API key → `401 Unauthorized`
   - Ижил `messageId`-г дахин илгээх → `200 duplicate` (давхар орохгүй)
4. Жишээ curl тест өгөгдөл:
   ```bash
   curl -X POST http://localhost:PORT/api/transactions \
     -H "Content-Type: application/json" \
     -H "X-API-Key: <WEBSITE_API_KEY>" \
     -d '{"messageId":"test-1","amount":14412.34,"currency":"MNT","date":"2026-06-07","description":"2266 NetflMCI","type":"expense","accountLast4":"50"}'
   ```
5. DB-д бичигдсэн эсэхийг шалга.

## Алхам 4: Хуучин имэйлээс тайлбар цуглуулах

1. `node scripts/collect-descriptions.js` ажиллуул.
2. Гаралтыг шалга: `descriptions-summary.csv` (давтамжаар эрэмбэлсэн), `transactions-export.json`.
3. Хэрэглэгчид үр дүнг харуул: нийт имэйл тоо, амжилттай задлагдсан тоо, хамгийн их давтагддаг тайлбарууд.
4. Хэрэв олон имэйл "parse_failed" болвол — parser-ийн log-г шалж, ямар формат задлагдаагүйг хэрэглэгчид мэдэгд (Голомт банк зарим имэйлээ өөр форматтай илгээж магадгүй).

## Алхам 5: Categorize дүрмийг бодит өгөгдөлд тааруулах

1. Алхам 4-ийн `descriptions-summary.csv`-г уншиж, хамгийн их давтагддаг мерчантуудыг тодорхойл.
2. Хэрэглэгчтэй хамтран (эсвэл хэрэглэгчид санал болгож) эдгээрийг `categories.config.js`-ийн тохирох ангилалд keyword болгож нэмэх.
   - Жишээ: `2266 NetflMCI` олон бол → "Захиалга / Сервис"-д `netfl` keyword нэмэх.
3. `categorize.test.js`-ийг бодит жишээгээр шинэчилж, дахин ажиллуулж зөв ангилж байгааг шалга.
4. `transactions-export.json` дээр categorize ажиллуулж, хэдэн хувь нь "Бусад" биш зөв ангилагдаж байгааг хэрэглэгчид харуул.

## Алхам 6: End-to-end тест (listener бүх урсгал)

1. API асаалттай байгаа эсэхийг баталгаажуул.
2. Listener-ийг асаа (`node index.js` эсвэл шууд).
3. Listener Gmail-д амжилттай холбогдож, IDLE горимд орсон log-г шалга.
4. End-to-end тест: дараах хоёрын аль нэгээр:
   - **a)** Хэрэглэгч өөртөө жижиг бодит гүйлгээ хийж, Голомтоос имэйл ирэхийг хүлээх, эсвэл
   - **b)** Хадгалсан жишээ имэйлийг (collect-descriptions-ийн export-оос) дахин боловсруулах test mode.
5. Урсгал бүрэн ажилласныг баталгаажуул: имэйл барьсан → parse → categorize → API → DB. Алхам бүрийн log-г харуул.
6. Идэмпотентность шалга: ижил имэйл хоёр удаа боловсруулагдвал DB-д давхар орохгүй.

## Алхам 7: Найдвартай ажиллагааг баталгаажуулах

1. Token refresh ажиллаж байгааг шалга (50 минут хүлээх боломжгүй тул код дотор timer зөв тохируулагдсан эсэхийг review хий).
2. Reconnect тест: listener ажиллаж байх үед сүлжээ/холболтыг түр тасалж, автоматаар дахин холбогдож байгааг шалга.
3. `lastSeenUid` DB-д хадгалагдаж, restart хийсний дараа алгасахгүй байгааг шалга.
4. `pm2`-оор асаах тохиргоог баталгаажуул: `pm2 start ecosystem.config.cjs`, дараа `pm2 logs`-оор log урсаж байгааг харуул.

## Гүйцэтгэлийн дараа

Бүх алхам амжилттай болсны дараа хэрэглэгчид товч тайлан гарга:
- Token авсан эсэх
- API тест (201/401/duplicate) үр дүн
- Цуглуулсан имэйл тоо, ангилагдсан хувь
- End-to-end тест амжилттай эсэх
- Дараа хийх зүйл (VPS deployment, frontend dashboard)

## Чухал тэмдэглэл агентад

- Алхмуудыг ДАРААЛЛААР гүйцэтгэ. Нэг алхам бүтэлгүй бол дараагийнх руу бү оро — хэрэглэгчид асуудлыг тодорхой хэл.
- `get-token.js` болон бодит гүйлгээ хийх алхамд ХЭРЭГЛЭГЧИЙН оролцоо шаардлагатай — тэр хэсгийг тодорхой зааварчилж, хүлээ.
- Нууц утга (Client Secret, refresh token, API key) -г log-д хэвлэхгүй, дэлгэцэнд бүтнээр харуулахгүй.
- Алдаа гарвал бүтэн stack trace биш, ойлгомжтой тайлбар + шийдлийг хэл.
